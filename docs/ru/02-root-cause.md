# 2. Первопричина

Все офсеты ниже — это **файловые офсеты внутри распакованной PE32-секции**
драйвера `H2OFormBrowserDxe` (GUID модуля `9E5DAEB4-4B91-4466-9EBE-81C7E4401E6D`),
как он поставляется в NLCN37/38/39 (все три байт-в-байт одинаковы:
`sha256 2c36d96f8d6d20aa9f65084eb3cfe33b683fbf777439fc4a5954411d12d72502`).

## Инструкция-фильтр

`H2OFormBrowserDxe` сортирует и фильтрует список хэндлов HII FormSet перед тем,
как браузер отрисует вкладки. Его процедура сортировки (Ghidra `FUN_000314f4`,
база `0x314f4`) содержит такую последовательность:

```
офсет     байты              дизассемблер
--------  -----------------  -------------------------------------------
0x31709   ff 50 48           call qword ptr [rax+0x48]    ; FreePool
0x3170c   80 7b 10 00        cmp  byte ptr [rbx+0x10], 0  ; флаг FormSet
0x31710   74 de              jz   0x316f0                 ; флаг==0 -> пропуск
```

`[rbx+0x10]` — однобайтовый флаг «показывать этот FormSet» из внутренней таблицы.
Когда он `0`, `jz` уходит обратно в цикл **не добавляя FormSet в выходной список**
— FormSet молча выбрасывается.

## Таблица флагов

Внутренняя таблица находится по офсету `0x37c30` (записи по 20 байт: 16-байтовый
GUID FormSet + 4-байтовый флаг). Значимые строки:

| FormSet | GUID | флаг |
|---|---|---|
| Debug | `A6E38A2F-…` | 0 |
| Main | `72E7C0DF-…` | 1 |
| Security | `7A3A87F5-…` | 1 |
| Boot | `7B3A3A54-…` | 1 |
| (чипсет) | `C1E0B01A-…` | 0 |
| **Advanced** | **`C6D4769E-7F48-4D2A-98E9-87ADCCF35CCC`** | **0** |
| (прочее) | `5204F764-…` | 1 |
| Power | `A6712873-…` | 0 |
| (прочее) | `2D068309-…` | 1 |
| (прочее) | `B863B959-…` | 0 |
| CBS | `BFB68FED-…` | 0 |
| (прочее) | `B6936426-…` | 1 |

Флаг FormSet Advanced равен `0`, поэтому `jz` на `0x31710` убирает его из каждой
сессии браузера.

**Ключевой вывод:** значение флага **не** менялось между версиями прошивки —
Advanced всегда был `0`. Изменилось то, что *сама инструкция-фильтр* была
добавлена. В сборках до NLCN37 в процедуре сортировки нет
`cmp byte [reg+0x10],0 ; jz`, поэтому FormSet с флагом = 0 никогда не
пропускались и Advanced был виден. См. [матрицу версий](05-cross-version-matrix.md)
для точной границы (NLCN36 → NLCN37).

## Путь вызова (почему штатная утилита попадает на это)

`SetupUtilityApp` строит список вкладок и вызывает браузер обычным способом:

```
SetupUtilityApp
  -> EFI_FORM_BROWSER2_PROTOCOL.SendForm(handles, count, FormSetGuid=NULL, FormId=1, …)
       -> внутренний SendForm H2OFormBrowserDxe (FUN_00001018)
            -> FUN_000314f4   ; сортировка + ФИЛЬТР  <-- здесь выбрасываются FormSet с флагом==0
            -> построение списка страниц из выживших хэндлов
            -> отрисовка вкладок
```

`SendForm` вызывается с **NULL/нулевым FormSetGuid** (что значит «покажи все
переданные хэндлы») — это штатное поведение. Список хэндлов, который передаёт
`SetupUtilityApp`, *включает* хэндл HII FormSet Advanced — именно
`FUN_000314f4` внутри браузера его выбрасывает. Поэтому патчинг
`SetupUtilityApp` (переупорядочивание или урезание её списка хэндлов) никогда не
работал: фильтр срабатывает после, внутри браузера, независимо от этого.

## Исправление

Заменить 2-байтовый условный переход на два `NOP`:

```
офсет 0x31710:   74 de   (jz 0x316f0)      ->   90 90   (nop; nop)
```

Теперь `cmp byte [rbx+0x10],0` всё ещё выполняется, но его результат
игнорируется; управление проваливается дальше и **каждый** FormSet — включая
Advanced (а также CBS, Debug и т. д.) — добавляется в список. Это в точности
воспроизводит поведение до NLCN37.

Больше ничего не трогается: таблица флагов не изменена, `SendForm` по-прежнему
вызывается с нулевым FormSetGuid, список хэндлов не изменён, а `SetupUtilityApp`
— байт-в-байт штатный бинарь Lenovo.

## Почему детектор обобщённый, а не по фиксированным байтам

Точные байты (`ff 50 48 80 7b 10 00 74 de`) специфичны для этого вывода
компилятора. На другой сборке могут отличаться регистр (`rbx`), смещение перехода
(`0xde`) или предшествующая инструкция. Семантически инвариантна сама **идиома**:

```
cmp byte ptr [<любой базовый регистр>+0x10], 0   ;  80 /7 [mod=01] 10 00
j(n)z <назад>                                     ;  74|75 <отрицательный rel8>
```

И рантайм-патчер, и оффлайн-анализатор сопоставляют эту идиому структурно,
поэтому продолжают работать между версиями и на схожих моделях с тем же кодом
браузера форм. См. [воспроизведение анализа](03-reproduce-analysis.md).
